# Глубокий разбор conversation: Virtuoso -> TanStack Virtual

Этот документ - большой технический разбор того, как устроен conversation в workspace.

Его задача - не просто перечислить файлы, а восстановить полную архитектуру:

1. как работало старое решение на Virtuoso;
2. что именно там виртуализировалось;
3. как работал и работает `useConversationHistory`;
4. как устроено текущее решение на TanStack Virtual;
5. почему произошла миграция;
6. какие файлы были добавлены и зачем;
7. что сейчас делает hybrid virtualization;
8. что старый компонент давал нам "из коробки", а что теперь мы контролируем сами.

## Что именно мы разбираем

Документ касается только conversation list в workspace.

Он **не** покрывает:

- отдельную виртуализацию в File Diff;
- отдельные process logs viewers вне main conversation;
- другие списки в приложении.

Главный pipeline, который здесь разбирается:

```text
execution processes
-> useConversationHistory
-> PatchTypeWithKey[]
-> aggregation / filtering
-> render model / virtualization
-> scroll behaviour
```

## Словарь терминов

### `ExecutionProcess`

Backend-сущность, которая представляет единицу выполнения. Это может быть:

- coding agent turn;
- follow-up turn;
- review turn;
- setup script;
- cleanup script;
- archive script;
- tool install script.

### `PatchTypeWithKey`

Плоская entry-модель, которую эмитит `useConversationHistory`.

Исторически это был основной тип conversation item до слоя aggregation.

### `DisplayEntry`

UI-уровень conversation item после aggregation.

Это может быть:

- обычная patch entry;
- aggregated tool group;
- aggregated diff group;
- aggregated thinking group.

### `ConversationRow`

Новая semantic row model, которая была добавлена в TanStack-архитектуре поверх `DisplayEntry`.

### `head`

Виртуализированная часть текущего conversation списка.

### `tail`

Невиртуализированная часть текущего conversation списка, которая рендерится обычным DOM.

## Короткий итог в двух абзацах

Старое решение использовало `@virtuoso.dev/message-list` как специализированный chat/message-list компонент. Оно получало не просто массив данных, а объект `data + scrollModifier`, само мерило элементы, само управляло message-list semantics и само выполняло большую часть scroll logic.

Текущее решение оставило `useConversationHistory` как источник conversation semantics, но заменило list engine на явную композицию:

- semantic row model;
- TanStack Virtual hook;
- explicit scroll intent model;
- separate scroll executor;
- hybrid virtualization (`virtualized head + unvirtualized tail`);
- явный empty state и anchor-preservation.

То есть миграция была не просто заменой библиотеки, а сменой архитектурной философии: от black-box message-list к app-owned virtualization architecture.

## Схема старой и новой архитектуры

### Старое решение на Virtuoso

```text
useExecutionProcessesContext()
        |
        v
useConversationHistory()
        |
        | emits: PatchTypeWithKey[] + AddEntryType + loading
        v
ConversationListContainer
        |
        | aggregateConsecutiveEntries(...)
        | filter next_action / token_usage_info
        | choose ScrollModifier
        v
setChannelData({ data, scrollModifier })
        |
        v
VirtuosoMessageList<DisplayEntry>
        |
        v
ItemContent(...)
        |
        v
DisplayConversationEntry / NewDisplayConversationEntry
```

### Текущее решение на TanStack Virtual

```text
useExecutionProcessesContext()
        |
        v
useConversationHistory()
        |
        | emits: PatchTypeWithKey[] + AddEntryType + loading
        v
ConversationListContainer
        |
        | aggregateConsecutiveEntries(...)
        | filter non-renderable entries
        | buildConversationRowsIncremental(...)
        | split into virtualized head + DOM tail
        | resolve scroll intent
        v
useConversationVirtualizer(rows=head)
        + useScrollCommandExecutor(...)
        |
        v
renderRowContent(...)
        |
        v
DisplayConversationEntry
```

Главное отличие:

- раньше list engine был message-list abstraction;
- теперь list engine - это набор наших собственных слоев поверх generic virtualizer.

## Старое решение на Virtuoso

## Какая библиотека использовалась

Старый conversation использовал:

- `@virtuoso.dev/message-list`

Это важно подчеркнуть отдельно:

- это не обычный `react-virtuoso` list;
- это коммерческий message-list API, специально заточенный под chat-like use case.

Ключевая особенность этого API:

- список принимал `DataWithScrollModifier<T>`, а не просто `T[]`.

То есть data update и scroll instruction шли в список одним пакетом.

## Что именно виртуализировалось в старом решении

Старый Virtuoso list виртуализировал **всю** conversation timeline как один flat list `DisplayEntry[]`.

В этот список входили:

- user messages;
- assistant messages;
- system messages;
- thinking messages;
- error messages;
- tool summary rows;
- plans;
- todo rows;
- subagent/task rows;
- script rows;
- file-edit rows;
- aggregated tool groups;
- aggregated diff groups;
- aggregated thinking groups.

Не являлись обычными items списка:

- setup placeholder;
- cleanup placeholder.

Они жили в `Header` и `Footer` самого Virtuoso message-list.

То есть old layout model был такой:

```text
Header
  + setup placeholder if needed

Virtuoso items
  + every conversation row as a flat DisplayEntry

Footer
  + cleanup placeholder if needed
```

## Старый `ConversationListContainer`: центральная идея

Самое важное состояние старого container-а:

```tsx
const [channelData, setChannelData] =
  useState<DataWithScrollModifier<DisplayEntry> | null>(null);
```

Это главный архитектурный ключ старой системы.

Старый container не делал так:

```tsx
setEntries(newEntries);
```

Он делал так:

```tsx
setChannelData({
  data: filteredEntries,
  scrollModifier,
});
```

То есть при каждом update он одновременно решал:

1. какие items показать;
2. как после этого должен вести себя scroll.

## Старый Virtuoso container: основной код

Ниже - главный старый render block из pre-swap container-а:

```tsx
<VirtuosoMessageList<DisplayEntry, MessageListContext>
  ref={messageListRef}
  className="h-full scrollbar-none"
  data={channelData}
  initialLocation={INITIAL_TOP_ITEM}
  context={messageListContext}
  computeItemKey={computeItemKey}
  itemIdentity={itemIdentity}
  ItemContent={ItemContent}
  onScroll={handleScroll}
  Header={({ context }) => (
    <div className="pt-2">
      {context?.showSetupPlaceholder && (
        <div className="my-base px-double">
          <ChatScriptPlaceholder
            type="setup"
            onConfigure={context.onConfigureSetup}
          />
        </div>
      )}
    </div>
  )}
  Footer={({ context }) => (
    <div className="pb-2">
      {context?.showCleanupPlaceholder && (
        <div className="my-base px-double">
          <ChatScriptPlaceholder
            type="cleanup"
            onConfigure={context.onConfigureCleanup}
          />
        </div>
      )}
    </div>
  )}

```

### Что означал каждый важный prop

- `data={channelData}`
  - объект вида `items + scroll instruction`.
- `initialLocation={INITIAL_TOP_ITEM}`
  - старт у нижнего края списка.
- `context={messageListContext}`
  - контекст, который попадал в `ItemContent`, `Header`, `Footer`.
- `computeItemKey`
  - React key.
- `itemIdentity`
  - identity для самого Virtuoso.
- `ItemContent`
  - единый renderer каждого item.
- `onScroll={handleScroll}`
  - механизм old at-bottom detection.

## Старые scroll modifiers

Из `packages/web-core/src/shared/lib/virtuoso-modifiers.ts` и старого container-а:

```ts
export const INITIAL_TOP_ITEM = {
  index: "LAST" as const,
  align: "end" as const,
};

export const InitialDataScrollModifier: ScrollModifier = {
  type: "item-location",
  location: INITIAL_TOP_ITEM,
  purgeItemSizes: true,
};

export const ScrollToBottomModifier: ScrollModifier = {
  type: "item-location",
  location: INITIAL_TOP_ITEM,
};

const AutoScrollToBottom: ScrollModifier = {
  type: "auto-scroll-to-bottom",
  autoScroll: "smooth",
};

const ScrollToTopOfLastItem: ScrollModifier = {
  type: "item-location",
  location: {
    index: "LAST",
    align: "start",
  },
};
```

### Что это значило по смыслу

- `InitialDataScrollModifier`
  - initial data load;
  - jump to bottom;
  - плюс `purgeItemSizes`, то есть сбросить cached item sizes Virtuoso.
- `ScrollToBottomModifier`
  - обычный jump to bottom без purge.
- `AutoScrollToBottom`
  - режим live follow для streaming.
- `ScrollToTopOfLastItem`
  - special case для plan presentation.

## Старый `ItemContent`: как рендерился каждый элемент

Старый render path был централизован в `ItemContent`:

```tsx
const ItemContent: VirtuosoMessageListProps<
  DisplayEntry,
  MessageListContext
>["ItemContent"] = ({ data, context }) => {
  const attempt = context?.attempt;
  const resetAction = context?.resetAction;

  if (isAggregatedGroup(data)) {
    return (
      <DisplayConversationEntry
        expansionKey={data.patchKey}
        aggregatedGroup={data}
        aggregatedDiffGroup={null}
        aggregatedThinkingGroup={null}
        entry={null}
        executionProcessId={data.executionProcessId}
        workspaceWithSession={attempt}
        resetAction={resetAction}
      />
    );
  }

  if (isAggregatedDiffGroup(data)) {
    return (
      <DisplayConversationEntry
        expansionKey={data.patchKey}
        aggregatedGroup={null}
        aggregatedDiffGroup={data}
        aggregatedThinkingGroup={null}
        entry={null}
        executionProcessId={data.executionProcessId}
        workspaceWithSession={attempt}
        resetAction={resetAction}
      />
    );
  }

  if (data.type === "NORMALIZED_ENTRY" && attempt) {
    return (
      <DisplayConversationEntry
        expansionKey={data.patchKey}
        entry={data.content}
        aggregatedGroup={null}
        aggregatedDiffGroup={null}
        aggregatedThinkingGroup={null}
        executionProcessId={data.executionProcessId}
        workspaceWithSession={attempt}
        resetAction={resetAction}
      />
    );
  }

  return null;
};
```

### Ключевая идея

Virtuoso сам по себе ничего не знал о user message, plan, diff group или approval.

Он знал только flat list of `DisplayEntry`.

Вся semantic dispatch logic жила в `DisplayConversationEntry` / `NewDisplayConversationEntry`.

То есть old division of responsibility был таким:

- Virtuoso отвечает за virtualization + scroll;
- renderer отвечает за semantic render конкретного item;
- `useConversationHistory` отвечает за то, какие items вообще существуют.

## Старый identity model

Ключи были такими:

```tsx
const computeItemKey = ({ data }) => `conv-${data.patchKey}`;
const itemIdentity = (item) => item.patchKey;
```

Это означает:

- React key = `conv-${patchKey}`;
- identity для Virtuoso = `patchKey`.

`patchKey` был центральной осью identity всего старого conversation pipeline.

## Старый `useConversationHistory`: его роль

Даже в старой архитектуре источник данных был не Virtuoso, а именно `useConversationHistory`.

Хук делал четыре главные вещи:

1. читал execution processes из `useExecutionProcessesContext()`;
2. подгружал historic entries для completed processes;
3. стримил live entries для running process;
4. превращал все это в единый плоский conversation stream.

То есть старая архитектура уже была логически разделена на:

- data semantics -> `useConversationHistory`;
- render/scroll engine -> `ConversationListContainer` + Virtuoso.

## Старый `flattenEntriesForEmit(...)`: главное semantic ядро

Самый важный старый semantic слой был в `flattenEntriesForEmit(...)`.

Он делал следующее:

- создавал synthetic `user_message` для coding/review processes;
- вырезал raw `user_message` из process log, чтобы не было дублей;
- вырезал `token_usage_info` из видимого conversation;
- определял pending approval;
- определял running process;
- добавлял synthetic `loading` patch;
- превращал script processes в synthetic `tool_use` row;
- добавлял `next_action`, когда ничего больше не running.

Representative old code:

```ts
if (
  p.executionProcess.executor_action.typ.type === "CodingAgentInitialRequest" ||
  p.executionProcess.executor_action.typ.type ===
    "CodingAgentFollowUpRequest" ||
  p.executionProcess.executor_action.typ.type === "ReviewRequest"
) {
  const userNormalizedEntry: NormalizedEntry = {
    entry_type: {
      type: "user_message",
    },
    content: actionType.prompt,
    timestamp: null,
  };

  const userPatch: PatchType = {
    type: "NORMALIZED_ENTRY",
    content: userNormalizedEntry,
  };

  entries.push(userPatchTypeWithKey);

  const entriesExcludingUser = p.entries.filter(
    (e) =>
      e.type !== "NORMALIZED_ENTRY" ||
      (e.content.entry_type.type !== "user_message" &&
        e.content.entry_type.type !== "token_usage_info"),
  );

  entries.push(...entriesExcludingUser);

  if (isProcessRunning && !hasPendingApprovalEntry) {
    entries.push(makeLoadingPatch(p.executionProcess.id));
  }
}
```

Это один из самых важных фактов во всей системе:

**conversation list рендерил не сырые process logs, а намеренно построенный semantic stream.**

## Старый initial load

Старый initial load работал так:

1. completed processes проходились в reverse order;
2. из них загружались historical entries;
3. загрузка останавливалась, как только accumulated entries превышали `MIN_INITIAL_ENTRIES`;
4. дальше conversation эмитился как `'initial'`;
5. container выбирал `InitialDataScrollModifier`;
6. Virtuoso прыгал в низ и purge-ил cached sizes.

Representative old code из hook:

```ts
for (const executionProcess of [...executionProcesses.current].reverse()) {
  if (executionProcess.status === ExecutionProcessStatus.running) continue;

  const entries =
    await loadEntriesForHistoricExecutionProcess(executionProcess);
  const entriesWithKey = entries.map((e, idx) =>
    patchWithKey(e, executionProcess.id, idx),
  );

  localDisplayedExecutionProcesses[executionProcess.id] = {
    executionProcess,
    entries: entriesWithKey,
  };

  if (
    flattenEntries(localDisplayedExecutionProcesses).length >
    MIN_INITIAL_ENTRIES
  ) {
    break;
  }
}
```

И затем old container выбирал scroll modifier:

```ts
if (loading) {
  scrollModifier = InitialDataScrollModifier;
} else if (pending.addType === "plan") {
  scrollModifier = ScrollToTopOfLastItem;
} else if (pending.addType === "running") {
  scrollModifier = AutoScrollToBottom;
} else {
  scrollModifier = ScrollToBottomModifier;
}
```

## Старый use case: добавление нового prompt

Это один из самых важных сценариев для понимания старой архитектуры.

### Сценарий

Пользователь пишет второй prompt в уже существующем conversation.

### Что происходило по шагам

1. Backend создает новый `ExecutionProcess` c `CodingAgentFollowUpRequest`.
2. `useExecutionProcessesContext()` получает новый process.
3. `useConversationHistory` видит новый running process.
4. `ensureProcessVisible(activeProcess)` создает пустую оболочку process state.
5. `emitEntries(..., 'running', false)` немедленно перестраивает conversation.
6. Уже на этом шаге conversation может содержать:
   - synthetic user row;
   - synthetic loading row.
7. Container получает новый `entries + addType`.
8. Через debounce формируется `setChannelData({ data, scrollModifier })`.
9. Если `addType === 'running'`, выбирается `AutoScrollToBottom`.
10. Virtuoso сам выполняет bottom-follow.

Representative old code:

```ts
for (const activeProcess of activeProcesses) {
  if (!displayedExecutionProcesses.current[activeProcess.id]) {
    const runningOrInitial =
      Object.keys(displayedExecutionProcesses.current).length > 1
        ? "running"
        : "initial";

    ensureProcessVisible(activeProcess);
    emitEntries(displayedExecutionProcesses.current, runningOrInitial, false);
  }

  if (
    activeProcess.status === ExecutionProcessStatus.running &&
    !streamingProcessIdsRef.current.has(activeProcess.id)
  ) {
    streamingProcessIdsRef.current.add(activeProcess.id);
    loadRunningAndEmitWithBackoff(activeProcess).finally(() => {
      streamingProcessIdsRef.current.delete(activeProcess.id);
    });
  }
}
```

### Что видел пользователь

Еще до прихода полноценного assistant output пользователь мог увидеть:

```text
...предыдущий conversation...
You: <новый prompt>
Loading...
```

Это происходило потому, что хук сам синтезировал эти rows.

## Старый streaming flow

Когда live stream стартовал, хук обновлял entries процесса на каждом streamed patch batch:

```ts
const controller = streamJsonPatchEntries<PatchType>(url, {
  onEntries(entries) {
    const patchesWithKey = entries.map((entry, index) =>
      patchWithKey(entry, executionProcess.id, index),
    );
    mergeIntoDisplayed((state) => {
      state[executionProcess.id] = {
        executionProcess,
        entries: patchesWithKey,
      };
    });
    emitEntries(displayedExecutionProcesses.current, "running", false);
  },
  onFinished: () => {
    emitEntries(displayedExecutionProcesses.current, "running", false);
    controller.close();
    resolve();
  },
});
```

Дальше container eventually делал:

```ts
setChannelData({ data: filteredEntries, scrollModifier: AutoScrollToBottom });
```

И Virtuoso сам продолжал follow output.

## Важный недостаток старого debounce

В старом container-е обновления шли через trailing debounce:

```ts
debounceTimeoutRef.current = setTimeout(() => {
  ...
}, 100);
```

Но сам upstream stream уже приходил достаточно часто.

Поэтому при непрерывном streaming этот timeout мог постоянно сбрасываться. То есть старый update model был менее детерминированным, чем казался на первый взгляд.

## Старый flow при завершении процесса

Когда running process заканчивался, old hook не считал streamed snapshot окончательной truth.

Он перезагружал canonical historical entries этого process:

```ts
if (
  previousStatus === ExecutionProcessStatus.running &&
  currentStatus !== ExecutionProcessStatus.running &&
  displayedExecutionProcesses.current[process.id]
) {
  processesToReload.push(process);
}
```

Дальше:

```ts
const entries = await loadEntriesForHistoricExecutionProcess(process);
mergeIntoDisplayed((state) => {
  state[process.id] = {
    executionProcess: process,
    entries: entriesWithKey,
  };
});
emitEntries(displayedExecutionProcesses.current, "running", false);
```

То есть lifecycle был такой:

- пока процесс running -> render from live stream;
- когда процесс finished -> replace with canonical history.

## Что Virtuoso давал нам "из коробки"

Это важный список, потому что именно его теперь пришлось декомпозировать на отдельные файлы.

### 1. Measurement и height management

- автоматическое измерение variable-height rows;
- внутренний size cache;
- встроенную re-measure logic;
- `purgeItemSizes` для cold-start reset.

### 2. Message-list semantics

- специализированный chat component;
- `scrollToItem({ index: 'LAST' })` semantics;
- `getCurrentlyRendered()`;
- built-in scroll behaviour через message-list API.

### 3. Follow output

- режим `auto-scroll-to-bottom` был частью компонента;
- explicit app-owned scroll state machine не требовалась.

### 4. Whole-list virtualization

- весь conversation был одним виртуализированным списком;
- не было отдельного head/tail split.

## Почему старое решение стало проблемой

Старый компонент был удобен, но стратегически имел несколько серьезных ограничений.

### 1. Коммерческая зависимость

Система зависела от `VirtuosoMessageListLicense` и `VITE_PUBLIC_REACT_VIRTUOSO_LICENSE_KEY`.

### 2. Scroll и data были слишком сцеплены

Старый основной update выглядел так:

```ts
setChannelData({ data, scrollModifier });
```

Это делало трудным разделение двух вопросов:

- какие данные надо показать;
- как после этого надо скроллить.

### 3. Слишком много black-box поведения

Virtuoso сам скрыто контролировал:

- sizing;
- height cache;
- follow-bottom behaviour;
- часть item identity semantics;
- message-list scrolling lifecycle.

Это было удобно, но не очень удобно для эволюции продукта.

### 4. Сложно строить специальное поведение под conversation

Conversation у нас не обычный список.

Здесь есть:

- setup / cleanup scripts;
- synthetic user messages;
- synthetic loading rows;
- aggregation transitions;
- diff-heavy rows;
- approval cards;
- dynamic scrolling semantics;
- особые требования к newest turn.

Для такого списка app-owned architecture дает больше контроля, чем готовый black-box message-list.

## Текущая модель size estimation

Сейчас size estimation больше не скрыта внутри готового message-list.

Она задается явно:

```ts

  compact: 40,
  medium: 80,
  tall: 280,
  dynamic: 150,
  hidden: 0,
};
```

И дополняется width-aware estimate:

```ts
export function estimateSizeForRow(
  row: ConversationRow,
  containerWidthPx?: number | null,
): number {
  const base = SIZE_ESTIMATE_PX[row.estimationHint];

  if (
    containerWidthPx != null &&
    containerWidthPx > 0 &&
    containerWidthPx < NARROW_WIDTH_PX &&
    TEXT_HEAVY_FAMILIES.has(row.rowFamily)
  ) {
    return Math.round(base * NARROW_WIDTH_MULTIPLIER);
  }

  return base;
}
```

То есть current philosophy такая:

- coarse estimate принадлежит приложению;
- точная высота приходит из DOM measurement.

## Текущая scroll state machine

Сейчас scroll behaviour живет в `packages/web-core/src/features/workspace-chat/model/conversation-scroll-commands.ts`.

Representative code:

```ts
export function resolveScrollIntent(
  addType: AddEntryType,
  isInitialLoad: boolean,
  isAtBottom: boolean,
): ScrollIntent {
  if (isInitialLoad) {
    return { type: "initial-bottom", purgeEstimatedSizes: true };
  }

  if (addType === "plan") {
    return { type: "plan-reveal", align: "start" };
  }

  if (addType === "running") {
    return isAtBottom
      ? { type: "follow-bottom", behavior: "smooth" }
      : { type: "preserve-anchor" };
  }

  return isAtBottom
    ? { type: "follow-bottom", behavior: "auto" }
    : { type: "preserve-anchor" };
}
```

### Это одна из самых больших архитектурных разниц

Раньше scroll policy была зашита в выбор `ScrollModifier`, который уходил прямо в Virtuoso data API.

Теперь scroll policy - это явный app-owned state machine.

```ts
  _delta,
```

## Current hybrid virtualization

Это главный structural difference по сравнению со старой архитектурой.

### Старое решение

- одна virtualization engine на весь список.

### Текущее решение

- head виртуализируется;
- tail рендерится как обычный DOM.

Базовая константа:

```ts
const ALWAYS_UNVIRTUALIZED_TAIL_ROWS = 8;
```

Но реальный tail boundary определяется не только этой константой.

Representative code:

```tsx
const candidateFirstUnvirtualizedRowIndex = useMemo(() => {
  const firstTailRowIndex = Math.max(
    conversationRows.length - ALWAYS_UNVIRTUALIZED_TAIL_ROWS,
    0,
  );

  if (!hasActiveStreamingTurn) {
    return firstTailRowIndex;
  }

  for (let index = conversationRows.length - 1; index >= 0; index -= 1) {
    if (conversationRows[index]?.isUserMessage) {
      return Math.min(index, firstTailRowIndex);
    }
  }

  return firstTailRowIndex;
}, [conversationRows, hasActiveStreamingTurn]);
```

### Что это значит

Базовое правило:

- последние 8 rows всегда остаются в DOM tail.

Дополнительное правило для active turn:

- если идет streaming turn, tail расширяется назад до последнего `user_message`.

То есть весь current turn старается остаться в обычном DOM потоке.

## Tail pinning во время стрима

После отладки overlap/gap багов был добавлен pinning boundary.

Representative code:

```tsx
const streamingFirstUnvirtualizedRowIndex = useMemo(() => {
  const lastSettledTailStartIndex = lastSettledTailStartIndexRef.current;
  const fallbackPinnedTailStartIndex =
    lastSettledTailStartIndex == null
      ? candidateFirstUnvirtualizedRowIndex
      : Math.min(
          lastSettledTailStartIndex,
          candidateFirstUnvirtualizedRowIndex,
        );

  return pinnedTailStartIndex ?? fallbackPinnedTailStartIndex;
}, [candidateFirstUnvirtualizedRowIndex, pinnedTailStartIndex]);
```

И:

```tsx
useEffect(() => {
  if (!hasActiveStreamingTurn) {
    lastSettledTailStartIndexRef.current = candidateFirstUnvirtualizedRowIndex;
    setPinnedTailStartIndex(null);
    return;
  }

  setPinnedTailStartIndex((current) => {
    const base = current ?? lastSettledTailStartIndexRef.current;
    if (base == null) return candidateFirstUnvirtualizedRowIndex;
    return Math.min(base, candidateFirstUnvirtualizedRowIndex);
  });
}, [candidateFirstUnvirtualizedRowIndex, hasActiveStreamingTurn]);
```

### Зачем это понадобилось

Без pinning seam между virtualized head и unvirtualized tail мог двигаться во время active stream.

Это приводило к тому, что rows мигрировали из обычного DOM в virtualizer посреди turn-а. А virtualizer в этот момент еще мог иметь только cold estimates этих rows.

Это порождало:

- overlaps;
- вертикальные дыры;
- нестабильный autoscroll.

Pinning устраняет именно этот класс ошибок.

## Текущий `useConversationHistory`: что изменилось, а что нет

Это очень важный раздел.

### Что **не** изменилось концептуально

Хук все так же отвечает за:

- stream running entries;

- session-scoped reset/load semantics.

Representative current code:

## Сравнение по основным use case

## Use case 1: открытие существующего conversation

### Старый Virtuoso

1. hook подгружает enough completed processes;
2. эмитит `'initial'`;
3. container выбирает `InitialDataScrollModifier`;
4. Virtuoso скроллит в низ и purge-ит cached sizes.

### Текущий TanStack

1. hook reset-ится по `scopeKey`;
2. ждет readiness process stream;
3. эмитит initial entries;
4. container строит rows;
5. scroll intent становится `initial-bottom`;
6. scroll выполняется после measurement.

### Главное различие

Раньше:

- data и scroll были склеены.

Сейчас:

- data и scroll separate, но coordinated.

## Use case 2: первый prompt с setup script

Это важный кейс, потому что conversation order здесь нетривиален.

### Смысл поведения

Когда setup script существует, initial prompt не должен прыгать по списку из-за разных таймингов между script branch и coding-agent branch.

Representative current logic:

И потом:

```ts
  executionProcess?.status !== ExecutionProcessStatus.running

  ... emit initial prompt from script branch ...

```

То есть и старая, и новая архитектура стараются сохранить conversation order deterministic.

## Use case 3: второй prompt / follow-up prompt

### Старый Virtuoso

1. новый running process появляется;
2. `ensureProcessVisible(...)` добавляет process shell;
3. `flattenEntriesForEmit(...)` уже может показать:
   - new user message;
   - loading row;
4. container eventually выбирает `AutoScrollToBottom`;
5. Virtuoso продолжает follow output.

### Текущий TanStack

1. новый running process появляется;
2. hook так же синтезирует new user row и loading row;
3. container превращает entries в `ConversationRow[]`;
4. current-turn-aware tail расширяется назад до последнего user message;
5. tail boundary pin-ится во время streaming;
6. scroll intent решается явно:
   - `follow-bottom`, если пользователь у низа;
   - `preserve-anchor`, если пользователь читает историю.

### Главное улучшение

Теперь логика "следовать за стримом или нет" - это не побочный эффект library API, а explicit product decision.

## Use case 4: пользователь скроллит вверх, пока стрим продолжается

### Старый Virtuoso

Это в основном зависело от внутренней message-list semantics. Приложение только слушало `location.isAtBottom` через `onScroll`.

### Текущий TanStack

Это явно описано в `resolveScrollIntent(...)`:

```ts
if (addType === "running") {
  return isAtBottom
    ? { type: "follow-bottom", behavior: "smooth" }
    : { type: "preserve-anchor" };
}
```

То есть продуктовая политика теперь читается прямо из кода.

## Use case 5: процесс завершился

### Старое решение

- reload historical entries;
- replace process state;
- снова пройти через Virtuoso modifiers.

### Текущее решение

- reload historical entries;
- rebuild filtered entries;
- rebuild semantic rows;
- remeasure, если нужно;
- потом либо follow, либо preserve-anchor в зависимости от current scroll state.

## Use case 6: truly empty conversation

### Старое решение

Старое решение не имело такого четкого explicit empty-state surface.

### Текущее решение

Сейчас явно различаются:

- loading;
- empty conversation;
- non-empty conversation.

Это вышло из последующих hardening fixes.

## Главная мысль про `useConversationHistory`

Если свести весь документ к одному архитектурному тезису, то он такой:

`useConversationHistory` был и остается semantic heart conversation.

Поменялся list engine, но не изменился главный semantic смысл hook-а:

- какой execution process должен быть виден;
- какие entries надо построить synthetic образом;
- когда conversation считается running;
- когда надо показывать next action;
- как setup/cleanup scripts влияют на timeline.

### Общие обязанности hook-а в обеих архитектурах

- process discovery;
- historical loading;
- live streaming;
- flattening into visible conversation entries;
- synthetic user messages;
- synthetic loading rows;
- next-action semantics;
- token-usage extraction.

### Что поменялось внутри hook-а сильнее всего

Не semantic rules, а lifecycle correctness:

- explicit scope identity;
- reset-before-load;
- empty-state correctness;
- session-aware reset semantics.

## Итог

Миграция с Virtuoso на TanStack Virtual не была простой заменой библиотеки.

Это была архитектурная инверсия.

Старый подход говорил:

```text
дай message-list-движку данные и scroll instruction,
и он сделает основную работу сам
```

Текущий подход говорит:

```text
semantic conversation model принадлежит приложению,
scroll policy принадлежит приложению,
newest-turn strategy принадлежит приложению,
а virtualizer - это более низкоуровневый engine
```

Это больше кода.

Это больше ответственности.

Но именно поэтому текущую систему можно анализировать, развивать и чинить гораздо более осознанно, чем старую black-box message-list архитектуру.
