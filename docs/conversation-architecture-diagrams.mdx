# Схемы conversation architecture

Этот документ нужен как companion к аналитическим документам про conversation.

Его задача - не повторить весь текст, а дать несколько схем, которые помогают быстро увидеть, **где именно сейчас живет сложность** и **куда логичнее двигаться дальше**.

<Info>
Это не финальная архитектура и не implementation plan.

Это рабочие схемы для обсуждения.

</Info>

## Схема 1 — историческая Virtuoso architecture

```text
ExecutionProcesses stream
        |
        v
useConversationHistory
        |
        | PatchTypeWithKey[] + addType + loading
        v
ConversationListContainer
        |
        | aggregateConsecutiveEntries(...)
        | choose ScrollModifier
        v
DataWithScrollModifier<DisplayEntry>
        |
        v
VirtuosoMessageList
        |
        +--> Header (setup placeholder)
        +--> ItemContent(DisplayEntry)
        +--> Footer (cleanup placeholder)
```

### Сильная сторона этой схемы

- renderer получает и данные, и scroll behaviour как единый контракт.

### Слабая сторона этой схемы

- semantic pipeline и scroll contract сильнее привязаны к библиотеке.

## Схема 2 — текущая TanStack architecture

```text
ExecutionProcesses WS stream
        |
        v
ExecutionProcessesProvider
        |
        v
useConversationHistory
        |
        | PatchTypeWithKey[] + addType + loading
        v
ConversationListContainer
        |
        +--> aggregateConsecutiveEntries(...)
        +--> filter non-renderable entries
        +--> buildConversationRows(...)
        +--> resolve ScrollIntent
        +--> split rows into head / tail
                 |
                 +--> virtualized head -> TanStack Virtual
                 |
                 +--> unvirtualized tail -> plain DOM
```

### Сильная сторона этой схемы

- app-owned control над scroll, row model и rendering strategy.

### Слабая сторона этой схемы

- semantic complexity размазана между hook, aggregation layer, row model и container orchestration.

## Схема 3 — sketch с созвона: что в нем полезно, а что нет

На скриншоте с созвона видны две разные интуиции.

### Верхняя версия sketch

По смыслу она выглядит примерно так:

```text
WebSocket to get execution processes
-> list of execution processes
-> split:
     running process -> fetch it
     previous execution process -> fetch it
-> update conversation
```

### Что в ней полезного

- она честно показывает, что conversation не является прямым API response;
- она показывает, что execution process list и conversation - не одно и то же.

### Что в ней слабого

- она слишком imperative;
- в ней нет отдельного semantic derivation layer;
- в ней не показан render/virtualization layer;
- создается впечатление, что conversation можно просто собрать из пары fetch steps.

### Нижняя версия sketch

По смыслу она выглядит ближе к такому:

```text
WebSocket to get execution processes for a session
-> handler
-> load historic execution process logs
-> load live execution process logs
-> given all available execution processes, render them
```

### Что в ней лучше

- появляется handler/orchestrator;
- признается разница между historic и live loading;
- conversation собирается не сразу, а через промежуточную processing layer.

### Что в ней все еще не хватает

- не отделен semantic timeline derivation от UI render layer;
- не видно aggregation/grouping слоя;
- не видно scroll model;
- не видно, где проходят границы between transport, semantics and renderer.

<Warning>
  Как рабочий sketch это полезно. Как финальная архитектурная схема - пока
  недостаточно точно.
</Warning>

## Схема 4 — более честная текущая decomposition

Если рисовать текущую систему более точно, она скорее выглядит так:

```text
Transport layer
  - session execution-process stream
  - per-process historic logs
  - per-process live logs

Semantic layer
  - useConversationHistory
  - synthetic user/loading/next_action/script rows
  - process lifecycle reconciliation

Post-semantic transform layer
  - aggregateConsecutiveEntries
  - filtering non-renderable items

Render model layer
  - ConversationRow
  - rowFamily / estimationHint / semanticKey

Render engine layer
  - TanStack Virtual head
  - DOM tail
  - scroll intent + executor
```

### Почему эта схема важна

Потому что она показывает: сегодня проблема уже не только в одном hook и не только в виртуализаторе.

Сложность распределена как минимум по 5 слоям.

## Схема 5 — рекомендуемое направление

На текущем этапе наиболее логично двигаться в такую сторону:

```text
Transport layer
  - execution-process streams
  - historic/live process logs

Timeline derivation layer   <-- главный следующий архитектурный фокус
  - turn ordering
  - synthetic rows
  - grouping semantics
  - script/setup/cleanup semantics
  - approval/edit/fork/stop semantics

Render model layer
  - stable semantic rows
  - row families
  - size classes

Render engine layer
  - TanStack Virtual or other engine
  - scroll policy
  - head/tail strategy
```

### Смысл этой схемы

- renderer перестает быть главным носителем semantic complexity;
- `useConversationHistory` перестает быть единственным overloaded orchestrator;
- выбор Virtuoso vs TanStack становится secondary decision, а не единственным архитектурным вопросом.

## Схема 6 — decision tree

```text
Is the main problem library-level scroll/measurement behaviour?
    |
    +-- partly yes
    |
    +-- but not only
            |
            v
Is semantic conversation derivation too coupled to renderer/orchestration?
            |
            +-- yes
                    |
                    v
Next strongest move:
  keep current renderer path,
  extract/clean semantic timeline boundary first,
  then re-evaluate renderer choice on a cleaner model
```
