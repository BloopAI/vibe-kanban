# Conversation architecture explained simply

Этот документ объясняет текущую conversation architecture простым языком.

Его цель — не заменить глубокие технические документы, а помочь быстро понять главную мысль:

<Check>
  Основная долгосрочная проблема conversation сейчас — не только в библиотеке
  виртуализации. Главная проблема в том, что слишком много разных задач смешано
  в нескольких слоях одновременно.
</Check>

## Если совсем коротко

Сейчас conversation-система делает сразу много разных вещей:

- получает сырые данные с backend;
- решает, какие из этих данных вообще должны стать частью conversation;
- добавляет synthetic blocks вроде `setup_script`, `clean_up_script`, `loading`, `next_action`;
- группирует некоторые entries;
- превращает это в rows для UI;
- виртуализирует их;
- управляет скроллом.

Когда все это сильно смешано, баги выглядят как “проблема виртуализации”, хотя часть причин на самом деле находится выше — в semantic pipeline.

## О каких слоях вообще идет речь

Ниже — упрощенная модель conversation architecture.

```text
Transport layer
  -> получает сырые execution processes и их live/historic logs

Timeline derivation layer
  -> решает, что считается conversation и в каком порядке это показывать

Render model layer
  -> превращает conversation в стабильные UI rows

Render engine layer
  -> рисует rows, виртуализирует их и скроллит список
```

## 1. Transport layer

Это самый нижний слой.

Он отвечает только за вопрос:

> Какие сырые данные у нас есть прямо сейчас?

Примеры:

- websocket со списком execution processes;
- historic logs отдельного process;
- live stream entries отдельного running process.

Этот слой не должен думать:

- что такое user message;
- куда вставить setup script;
- что считать текущим turn;
- когда надо скроллить вниз.

Он должен просто доставлять факты.

## 2. Timeline derivation layer

Это главный слой, который сейчас выглядит наиболее важным для улучшения.

Он отвечает на вопрос:

> Как из сырых backend-данных получается именно conversation, который видит пользователь?

Именно здесь должны жить такие решения:

- где появляется `user_message`;
- когда показать synthetic `loading` row;
- куда вставить `setup_script` и `clean_up_script`;
- как выглядят turn boundaries;
- как работают approvals и ask-user-question flows;
- как взаимодействуют `edit_message`, retry, stop, fork.

По сути это “мозг” conversation.

## 3. Render model layer

Этот слой нужен для того, чтобы renderer не получал сырые backend-shaped данные.

Он отвечает на вопрос:

> Какой набор стабильных rows мы реально рисуем на экране?

Каждый row здесь должен уже иметь:

- стабильный id/key;
- тип строки;
- связь с process/turn;
- примерный size hint;
- понятные свойства для scroll/navigation.

Это важный промежуточный слой между “смыслом conversation” и “отрисовкой conversation”.

## 4. Render engine layer

Это уже самый нижний UI-слой.

Он отвечает за:

- virtualizer (`TanStack` или `Virtuoso`);
- измерение высот;
- scroll-to-bottom;
- anchor preservation;
- head/tail strategy;
- layout stability.

То есть этот слой должен заниматься именно отображением и скроллом, а не смыслом conversation.

## В чем текущая проблема

Сейчас проблема не в том, что у нас “плохой virtualizer”.

Проблема в том, что semantic responsibility распределена сразу по нескольким местам:

- `useConversationHistory` строит conversation semantics;
- потом `aggregateConsecutiveEntries` еще раз меняет shape данных;
- потом `ConversationListContainer` еще раз принимает важные orchestration decisions;
- потом row model уже в UI layer решает, как это виртуализировать.

Из-за этого renderer часто получает не просто стабильный список rows, а систему, в которой сама форма данных продолжает меняться по дороге.

<Warning>
  Для любого virtualizer это тяжелые условия. Даже хороший renderer будет
  выглядеть нестабильно, если upstream timeline model сама по себе слишком
  сложная и подвижная.
</Warning>

## Почему проблема не сводится только к Virtuoso vs TanStack

На первый взгляд кажется, что вопрос простой:

- либо оставить Virtuoso;
- либо допилить TanStack.

Но на практике это не весь вопрос.

Если semantic layer перегружен, то:

- Virtuoso скроет часть проблем под более opinionated chat mechanics;
- TanStack даст больше контроля, но заставит нас явно увидеть все эти проблемы.

То есть renderer важен, но он не единственный источник сложности.

## Почему не хочется просто сразу вернуться на Virtuoso

Virtuoso действительно дает полезные встроенные механики:

- follow output;
- prepend anchoring;
- internal handling of dynamic heights;
- готовый message-list abstraction.

Но возврат к нему не решает главный вопрос:

> Где должна жить semantic complexity conversation?

Если оставить current semantic overload как есть, часть нестабильности останется — просто будет спрятана под более удобной библиотекой.

## Почему не хочется сразу переносить всё на backend

Идея backend-assisted timeline может быть сильной, но это уже следующий уровень.

Сначала полезнее самим понять:

- какая именно semantic model conversation нам нужна;
- что должно считаться turn;
- какие synthetic rows нам реально нужны;
- где проходят границы между process data и conversation timeline.

Если не прояснить это заранее, слишком ранний backend contract может зацементировать неправильную abstraction.

## Почему следующий шаг — отделить semantic timeline layer

Именно это сейчас выглядит наиболее разумным направлением.

Смысл такой:

1. не делать поспешный renderer swap;
2. не лечить только surface bugs;
3. сначала сделать более чистую и стабильную timeline model;
4. после этого уже честно сравнивать renderer-ы.

Это не означает “переписать всё заново”.

Это означает:

- уменьшить число мест, которые меняют смысл conversation;
- сделать renderer тоньше;
- сделать scroll layer менее зависимой от внутренней формы данных.

## Что это даст practically

Если это сделать, то:

- conversation станет более предсказуемой;
- renderer станет проще обсуждать и менять;
- баги будет легче локализовывать;
- будет понятнее, где проблема: в transport, в semantics или в render engine.

## Главная мысль документа

Если свести все к одной фразе, она такая:

<Check>
  Сейчас нам важнее не выбрать “идеальный virtualizer”, а сначала сделать так,
  чтобы conversation имел более чистую, стабильную и понятную архитектуру. После
  этого и TanStack, и Virtuoso можно будет оценивать гораздо честнее.
</Check>
